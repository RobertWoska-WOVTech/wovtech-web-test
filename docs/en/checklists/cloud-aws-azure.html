Cloud (AWS/Azure/GCP): what to think through before design

Cloud is not just hosting. It is an operating model and a set of rules that shape security, availability and cost. Before design, clarify standards, access, change control and recovery.

Cloud provides speed, but also introduces new risk when governance is weak. The minimum baseline is clear ownership, IAM + MFA, network boundaries, centralised logs and monitoring, IaC and tested recovery.

Key questions

  • Who owns accounts/subscriptions and who approves changes?
  • How are admin/root access, MFA and audit handled?
  • Do you have clear environment separation, naming, tagging and policy baselines?
  • Are changes made via IaC and pipelines or manually?
  • How is connectivity with on-prem, DNS and egress designed?
  • Are logs and monitoring centralised and are runbooks available?
  • Do you have defined RTO/RPO and tested recovery?
  • Do you have cost allocation, budgets and basic guardrails?

Practical starting point

Start with a minimum landing zone (access, logging, policy baseline, network boundaries), then introduce IaC and controlled change management. Expand services and automation afterwards.
