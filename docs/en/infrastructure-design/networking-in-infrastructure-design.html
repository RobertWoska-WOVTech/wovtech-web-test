Networking in infrastructure design: what must be clear

Networking is not a separate topic only for network engineers. It is a shared layer that directly affects availability, security and incident response.

Segmentation, firewall rules, access model and routing (on-prem ↔ cloud) must be named and documented – otherwise both risk and incident time grow.

Questions worth answering

  • What is the segmentation model and where are the trust boundaries?
  • Where are firewall boundaries, who owns them and what is the change process?
  • How is access handled (VPN/bastion/SSO, least privilege, time-limited access)?
  • How does routing work between on-prem and cloud?
  • Can you debug the network quickly (flow logs, trace and a standard triage procedure)?

Typical risks

  • Long incidents: without clear boundaries, triage becomes guesswork.
  • Security risk: temporary rules and exceptions stay forever.
  • Unclear ownership: routing/firewall/DNS issues bounce between teams.
